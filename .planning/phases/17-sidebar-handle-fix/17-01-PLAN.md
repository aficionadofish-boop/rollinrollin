Fix the root cause of the sidebar resize handle being immovable at 1100x750 standard launch size. Two independent fixes required:

**A. Deferred setSizes() via showEvent() in MainWindow (app.py):**

1. Add a `showEvent()` override to MainWindow:
   ```python
   def showEvent(self, event) -> None:
       super().showEvent(event)
       if not hasattr(self, '_splitter_initialized'):
           self._splitter_initialized = True
           if not self._sidebar.is_collapsed():
               sidebar_w = self._sidebar.get_expanded_width()
               total_w = self.width()
               self._main_splitter.setSizes([total_w - sidebar_w, sidebar_w])
   ```
   The guard flag `_splitter_initialized` prevents re-applying on subsequent show events (minimize/restore). The `is_collapsed()` check avoids fighting the collapsed state's fixed 24px allocation.

2. Remove the `setSizes()` block from `_load_persisted_data()` (app.py lines ~347-351). Keep the `set_expanded_width()` call so the width value is stored — just remove the `setSizes()` call since showEvent handles it. Specifically remove:
   ```python
   sidebar_w = self._sidebar.get_expanded_width()
   total_w = self.width()
   self._main_splitter.setSizes([total_w - sidebar_w, sidebar_w])
   ```
   Keep this line (it stores the value for showEvent to use later):
   ```python
   self._sidebar.set_expanded_width(self._current_settings.sidebar_width)
   ```

**B. Reduce sidebar minimumWidth in _expand() (encounter_sidebar.py):**

1. In `_expand()`, change `self.setMinimumWidth(200)` to `self.setMinimumWidth(150)`. This gives the splitter 50px more slack at 1100px (movable range goes from ~144px to ~194px).

2. In `resizeEvent()`, change the threshold `event.size().width() >= 200` to `event.size().width() >= 150` to match the new minimum.

3. Add an `is_collapsed()` accessor method to EncounterSidebarDock if one does not already exist:
   ```python
   def is_collapsed(self) -> bool:
       return self._collapsed
   ```

**C. Set object name on main_splitter (app.py):**

After the `_main_splitter` is created (line ~84), add:
```python
self._main_splitter.setObjectName("main_splitter")
```
This enables scoped CSS selectors (`QSplitter#main_splitter::handle`) so handle styling only affects the main sidebar splitter, not internal splitters in Library, Macro, or Combat tabs.

**Why these fixes work together:** The setSizes timing fix ensures the splitter respects the persisted width after Qt's layout pass. The reduced minimumWidth gives the splitter enough slack to allow dragging even at 1100px. The object name enables Task 2's scoped CSS.

**B. Hide handle when sidebar is collapsed (app.py):**

Update `_on_sidebar_collapse_toggled()` to toggle handle visibility:

```python
def _on_sidebar_collapse_toggled(self, collapsed: bool) -> None:
    """Adjust splitter sizes when sidebar collapses or expands."""
    if collapsed:
        total = sum(self._main_splitter.sizes())
        self._main_splitter.setSizes([total - 24, 24])
        self._main_splitter.setHandleWidth(0)
    else:
        total = sum(self._main_splitter.sizes())
        sidebar_w = self._sidebar.get_expanded_width()
        self._main_splitter.setSizes([total - sidebar_w, sidebar_w])
        self._main_splitter.setHandleWidth(9)
```

Adding `setHandleWidth(0)` when collapsed removes the grab zone entirely (no phantom clickable area next to the 24px strip). `setHandleWidth(9)` on expand restores it to match the CSS width. Note: the stylesheet `width: 9px` and `setHandleWidth(9)` may coexist — `setHandleWidth` controls the logical grab area while stylesheet controls rendering. Setting both to the same value avoids conflicts.

**Why not inline stylesheet override for collapsed state:** Using `setHandleWidth(0)` is simpler and more reliable than toggling inline stylesheets. The global scoped CSS handles the visual; the programmatic width handles the interaction area. When handleWidth is 0, there is nothing to render regardless of CSS.
