**BUG-03: CR change doesn't immediately cascade**

1. In `_on_cr_changed()` (or equivalent CR change handler), after calling `_rebuild_preview()` which recalculates the working copy:
   - Call `_sync_save_toggles()` to refresh the save checkboxes/spinboxes with new proficiency-based values.
   - Call a form repopulation method (e.g., `_populate_form()` or the specific section refresh methods) to update all editor form widgets (save spinboxes, skill spinboxes, attack to-hit labels) with the recalculated values from `_working_copy`.
   - Use `blockSignals(True)` on each spinbox before updating its value to prevent circular signal loops, then `blockSignals(False)` after.

**BUG-05: Ability score changes don't live-update dependent skills**

2. In `_on_ability_changed()` (or equivalent ability score change handler), BEFORE calling `_rebuild_preview()`:
   - Import or reference `SKILL_TO_ABILITY` mapping (from `src/domain/math_engine.py` or wherever it's defined — research says it covers 5 abilities, no CON).
   - For the changed ability (e.g., if INT changed), find all skills mapping to that ability (INT → Arcana, History, Investigation, Nature, Religion).
   - For each affected skill present in `self._working_copy.skills`:
     - Determine if the skill has proficiency (bonus == old_ability_mod + prof) or expertise (bonus == old_ability_mod + 2*prof) or just ability mod.
     - Recompute the skill bonus using the new ability modifier: `new_mod + (prof_factor * prof_bonus)`.
     - Update `self._working_copy.skills[skill_name]` with the new value.
   - Then call `_rebuild_preview()` which will display the updated skills.
   - Also update any skill UI spinboxes/labels in the editor form with `blockSignals(True)` around value changes.

3. If skills were added by the user (not present in original monster), they should also cascade when the underlying ability changes. The key is to check ALL entries in `_working_copy.skills` against the `SKILL_TO_ABILITY` mapping.

**BUG-04: Added skill proficiency not shown in modified color**

1. In `src/ui/monster_editor.py`, find the `_apply_highlights()` method. This method currently highlights AC, HP, ability scores, and saves with modified-value colors. Add skill highlighting logic:
   - Compare `self._working_copy.skills` against `self._original_monster.skills` (or the base monster skills).
   - For each skill in the working copy that differs from the original (either added or value changed), mark it as modified.
   - Pass this modified-skills information to the preview panel.

2. In `src/ui/monster_detail.py`, find the `show_monster()` method (or the section that renders skills). Currently skills are rendered as plain text. Change the skills rendering to:
   - Accept optional highlighted/modified skill names.
   - Wrap modified skill entries in a colored `<span>` using the same amber/steel-blue color convention used for other modified values (check the existing pattern — research says steel blue for equipment, amber for manual).
   - If the skill was user-added (not in original monster), use the manual-edit color (amber).
   - If the skill bonus changed due to equipment, use the equipment color (steel blue).

3. In `_apply_highlights()`, call `self._detail_panel.highlight_skills(modified_skills_dict)` (or equivalent method) to pass the highlighting data to the detail panel.

**BUG-07: Spellcasting foci not modifying DC/spell attack bonus**

4. In `src/ui/monster_editor.py`, find `_rebuild_preview()` (or the method that prepares the working copy for display).
   - Before calling `self._engine.recalculate(self._working_copy)`, check if `self._focus_bonus` is set (non-zero).
   - If `_working_copy` has `SpellcastingInfo` objects (e.g., in `_working_copy.spellcasting_info` or however spellcasting is stored), update each `SpellcastingInfo.focus_bonus` to `self._focus_bonus`.
   - This ensures the math engine and validator use the focus bonus when calculating spell attack = casting_mod + prof + focus, and spell save DC = 8 + casting_mod + prof + focus.

5. After recalculation, verify the preview panel displays the updated spell attack bonus and spell save DC reflecting the focus.

6. If the validator flags spell attack/DC as custom after focus is applied (because the raw Monster model doesn't have focus), make sure the validator accounts for focus_bonus in its expected-value calculation.
