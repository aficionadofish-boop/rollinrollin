**BUG-11: Stats toggle button does nothing**

The research identified that the code path _on_stats_menu → _apply_stat_visible_to_all → card.set_stat_visible() appears correct on paper, but the stats remain hidden. Investigate and fix:

1. Read `_on_stats_menu()` carefully. It uses `menu.exec()` which returns the triggered QAction. In PySide6, there may be a behavior difference where `menu.exec()` returns `None` even when a checkable action is toggled. If `selected is None`, the method returns early — nothing happens.

   **Fix approach A** (preferred): Instead of relying on `menu.exec()` return value, connect each QAction's `triggered(bool)` signal to a lambda/slot that updates `_visible_stats` and calls `_apply_stat_visible_to_all`. Then call `menu.exec()` just to show the menu — ignore its return value.

   ```python
   def _on_stats_menu(self):
       menu = QMenu(self)
       for stat_key, label in self._STAT_LABELS.items():
           action = QAction(label, menu)
           action.setCheckable(True)
           action.setChecked(self._visible_stats.get(stat_key, False))
           action.triggered.connect(
               lambda checked, key=stat_key: self._toggle_stat(key, checked)
           )
           menu.addAction(action)
       menu.exec(QCursor.pos())  # or wherever the menu is shown

   def _toggle_stat(self, stat_key: str, visible: bool):
       self._visible_stats[stat_key] = visible
       self._apply_stat_visible_to_all(stat_key, visible)
   ```

   **Fix approach B** (if A doesn't apply): If the current code uses `menu.exec()` return value, the issue might be that `exec()` returns `None` for checkable actions in PySide6. Switch to approach A.

2. Verify `_apply_stat_visible_to_all(stat_key, visible)` correctly iterates over all current cards and calls `card.set_stat_visible(stat_key, visible)`.

3. In `src/ui/combatant_card.py`, verify and fix `card.set_stat_visible(stat_key, visible)`: it must show/hide the individual stat label AND update `_stats_widget` container visibility based on `any(label.isVisible() for label in stat_labels)`. If `_stats_widget.setVisible(any_visible)` is missing after the individual label toggle, add it. This is the card-level half of BUG-11 — without it the container stays hidden even when individual stat labels are toggled visible.

4. Verify `_rebuild_cards()` re-applies `_visible_stats` to newly built cards via `_apply_stat_visible_to_all`. If it doesn't, add calls after card creation to apply current visibility state.

**BUG-15: Legendary resistances refill to max between saves**

1. Read `src/ui/app.py` and find all call sites of `reset_lr_counters()` on the SavesTab. The research identified that `_on_sidebar_encounter_changed()` calls `reset_lr_counters()`, and this handler is connected to `sidebar.encounter_changed` which fires on various sidebar interactions, not just actual membership changes.

2. Fix the reset logic. Two approaches:

   **Approach A** (recommended): In `_on_sidebar_encounter_changed()`, track the previous set of monster names. Only call `reset_lr_counters()` when the set of monster TYPES (unique monster names) changes, not when counts change.
   ```python
   def _on_sidebar_encounter_changed(self):
       current_names = set(name for name, count in self._sidebar.get_members())
       if current_names != self._prev_encounter_names:
           self._saves_tab.reset_lr_counters()
           self._prev_encounter_names = current_names
       # ... rest of handler
   ```
   Initialize `self._prev_encounter_names = set()` in the constructor.

   **Approach B**: Add a separate signal `encounter_members_changed` on the sidebar that only fires when monsters are added/removed (not count changes), and connect `reset_lr_counters()` to that signal instead.

3. In `src/ui/encounters_tab.py`, verify that `_execute_roll()` correctly preserves LR counters across rolls:
   - After detection returns `lr_max`, the code checks `if base_name in self._lr_counters:` and uses the persisted value.
   - When `_on_lr_used` is called (DM uses an LR), it decrements `self._lr_counters[base_name]`.
   - Between rolls (without encounter change), `self._lr_counters` should retain its values.
   If this logic is correct and only the reset was the problem, approach A is sufficient.

**BUG-16: 0 HP healthbar behavior**
