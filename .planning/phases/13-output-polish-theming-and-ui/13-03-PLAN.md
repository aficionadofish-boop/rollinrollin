<objective>
Extend the macro preprocessor to capture and expose key-value pairs from Roll20 `{{key=value}}` template fields. Currently, the preprocessor extracts `{{name=...}}` as template_name but discards all other key names — only keeping the values for expression evaluation. For template card rendering (Plan 05), the key names are essential for labeled rows.

Purpose: Without this change, a macro like `&{template:default}{{name=Fireball}}{{DC=15}}{{Type=DEX}}` would lose the labels "DC" and "Type", making it impossible to render a meaningful card. This is the critical data model gap identified in the research phase.

Output: CleanedMacro and MacroLineResult both gain a `template_fields: list[tuple[str, str]]` field. The preprocessor populates it during {{...}} extraction. The service propagates it through to the result.
</objective>

2. Modify `_extract_template_fields` method:
   - Change return type from `tuple[str, str | None]` to `tuple[str, str | None, list[tuple[str, str]]]`.
   - Add a `fields: list[tuple[str, str]] = []` accumulator.
   - In the `if "=" in content:` branch, AFTER the existing name check:
     - For non-name fields (`key.strip().lower() != "name"`): append `(key.strip(), value.strip())` to `fields`.
     - Keep the existing behavior of adding the value to `result_parts` for expression evaluation.
   - For bare values (`else` branch, no `=`): do NOT add to fields (these are expression-only).
   - Return `(cleaned, template_name, fields)` instead of `(cleaned, template_name)`.

3. Update `process_line` to handle the new 3-tuple return:
   Change:
   ```python
   line, template_name = self._extract_template_fields(line)
   ```
   To:
   ```python
   line, template_name, template_fields = self._extract_template_fields(line)
   ```
   And in the `return CleanedMacro(...)` call, add:
   ```python
   template_fields=template_fields,
   ```

IMPORTANT: The `template_fields` values are RAW — they may contain `[[...]]` inline roll tokens that haven't been resolved yet. For example, `{{damage=[[8d6]]}}` produces `("damage", "[[8d6]]")`. The value `"[[8d6]]"` is also kept in the expression for resolution. Plan 05 (TemplateCard) will need to match resolved inline_results to field values by token order.
